Read these files to understand the approval queue structure:
- /home/turin/getajob/approval_queue/main.py
- /home/turin/getajob/approval_queue/routes.py
- /home/turin/getajob/tests/conftest.py (to understand test patterns)

Then create a new test file at /home/turin/getajob/tests/test_approval_queue/test_routes.py with FastAPI TestClient tests.

The test file should test these behaviors:

1. **test_health_returns_ok** — GET /api/health returns 200 with {"status":"ok"}
2. **test_dashboard_redirects_when_unauthenticated** — GET / returns 307 (redirect to /login) without session cookie
3. **test_login_with_correct_password** — POST /login with correct password returns 302 (redirect) and sets session cookie
4. **test_login_with_wrong_password** — POST /login with wrong password stays on /login page (returns HTML, not redirect)
5. **test_dashboard_loads_when_authenticated** — POST /login with correct password, then GET / returns 200 with dashboard HTML
6. **test_api_stats_returns_data** — GET /api/stats (after login) returns JSON with state counts and total

Use these patterns:
- FastAPI TestClient from starlette.testclient
- Create a test instance of the app using the app factory from main.py (or import the app directly)
- For auth: POST /login with form data {"password": "..."} to get a session cookie
- Use httpx.AsyncClient or TestClient for sync tests
- Mock the database dependency to avoid needing a real DB

Key architectural note: The approval queue app has a middleware that checks for a session cookie. The /login endpoint validates against a configurable password. The dashboard and API endpoints return data from the database (or empty state if tables don't exist).

After creating the tests, run: python3 -m pytest tests/test_approval_queue/ -v --tb=short
Then run: python3 -m pytest tests/ -q --tb=short
